# AGENTS.md

本项目是一个 C++ 软光栅项目。后续所有功能开发都必须遵循“先学习对应图形学笔记，再对齐需求，再写文档，再开发，再交付手动测试，再等待 review 通过”的流程。

## 1. 开发前必做：先阅读 `computer-graphics-notes` 对应模块

每次开始一个新功能前，必须先根据功能主题定位并阅读 `computer-graphics-notes/` 中最相关的模块，再进入需求讨论。

建议按下面的主题映射选择笔记：

- `lesson01-project-overview-and-entry.md`：项目结构、入口、整体流程
- `lesson02-platform-window-and-input.md`：平台层、窗口、输入
- `lesson03-math-and-camera.md`：数学基础、坐标系、相机
- `lesson04-framebuffer-depth-and-msaa.md`：帧缓冲、深度、MSAA
- `lesson05-vertex-uniform-varyings-and-programmable-shader-interface.md`：顶点、Uniform、Varying、可编程接口
- `lesson06-software-rasterization-mainline.md`：软光栅主流程、三角形光栅化
- `lesson07-model-and-texture-resource-loading.md`：模型、纹理、资源加载
- `lesson08-from-blinn-phong-to-pbr.md`：光照模型、Blinn-Phong、PBR
- `lesson09-ibl-precomputation-and-toolchain.md`：IBL、预计算、工具链
- `lesson10-engineering-features-parallel-pipeline-imgui-and-profiling.md`：工程化、并行、调试 UI、性能分析

执行要求：

1. 先判断本次功能对应哪个 lesson。
2. 先阅读对应 lesson，再和用户讨论实现方案。
3. 回复时需要明确说明“本次已阅读的笔记文件”以及“这些笔记对当前功能的约束/启发”。
4. 如果一个功能跨多个模块，按主模块 + 相关模块一起阅读。
5. 如果找不到完全匹配的模块，至少阅读最接近的 lesson 和 `computer-graphics-notes/README.md`，并向用户说明映射依据。

## 2. 功能开发标准工作流

每次开发功能都必须严格按以下顺序执行，不能跳步。

### Step 1. 先和用户对齐需求

在写任何代码前，先和用户对齐需求。需要至少确认以下内容：

- 本次功能目标是什么
- 不做什么，边界在哪里
- 预期效果是什么
- 涉及哪些模块或渲染阶段
- 是否有接口、输入输出、资源格式或性能方面要求
- 用户如何判断这个功能算完成

要求：

- 未完成需求对齐前，不能直接开始写代码。
- 如果需求存在歧义，先提问澄清，再继续。
- 需求对齐完成后，需要给出一版清晰的需求摘要，等用户确认。

### Step 2. 需求对齐后先生成开发文档

用户确认需求后，先生成开发文档，再开始编码。

开发文档建议保存到类似路径：

- `docs/features/<feature-name>.md`
- 如果项目当下没有 `docs/` 目录，可以在首次需要时创建

开发文档至少应包含以下内容：

- 功能背景
- 需求摘要
- 对应阅读的 `computer-graphics-notes` 模块
- 技术方案与关键数据流
- 计划修改的文件
- 风险点与兼容性影响
- 手动测试方案
- 完成标准 / 验收标准

要求：

- 没有开发文档，不进入编码阶段。
- 如果实现过程中方案发生变化，需要同步更新开发文档。

### Step 3. 按开发文档实现代码

进入编码阶段后：

- 实现应与已确认需求和开发文档保持一致
- 尽量小步修改，避免引入与当前功能无关的改动
- 如果发现需求或方案存在问题，应先和用户同步，再决定是否调整

### Step 4. 编码完成后提供手动测试方法

代码完成后，必须明确告诉用户如何手动测试，至少包含：

- 如何编译 / 运行
- 需要准备的模型、纹理、场景或输入参数
- 操作步骤
- 预期现象 / 预期结果
- 如果结果异常，应该重点观察什么

要求：

- 手动测试说明要尽量可直接照着执行。
- 如果当前环境下无法实际运行测试，要明确说明未执行，并给出推荐的手测方式。

### Step 5. 等待用户 review，通过后才算本次功能结束

在提供代码与手动测试说明后，不要自行宣布功能结束，必须等待用户 review。

只有在用户明确表示以下含义之一时，才算本次功能开发完成：

- “review 通过”
- “可以了”
- “这个功能完成了”
- 其他等价确认

如果用户提出 review 修改意见，则继续当前功能迭代，流程为：

1. 先理解 review 意见并必要时再次对齐
2. 更新开发文档
3. 修改代码
4. 更新手动测试说明
5. 再次等待用户 review

## 3. 对话与执行规范

- 默认使用中文交流，除非用户要求英文。
- 每次开始新功能时，先说明本次准备阅读哪些 `computer-graphics-notes` 模块。
- 每次阶段切换时，明确当前处于哪个阶段：`阅读笔记`、`需求对齐`、`开发文档`、`编码实现`、`手动测试交付`、`等待 review`。
- 即使用户直接要求“先写代码”，也要先完成“阅读对应笔记 + 对齐需求 + 生成开发文档”。
- 未经用户 review 通过，不将该功能标记为完成。

## 4. 本项目的完成定义

一个功能只有同时满足以下条件，才算完成：

1. 已阅读并说明对应的 `computer-graphics-notes` 模块
2. 已与用户完成需求对齐
3. 已生成开发文档
4. 已完成代码实现
5. 已提供清晰的手动测试方法
6. 已通过用户 review

如果以上任一项未完成，则该功能仍处于进行中。
