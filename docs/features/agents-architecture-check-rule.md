# AGENTS 架构文档检查规则补充

## 功能背景

当前 `AGENTS.md` 已经要求：

- 开发前必须阅读架构图与对应图形学笔记
- 如果实现影响模块职责、依赖方向、关键数据流、共享核心数据结构，必须同步更新 `docs/architecture/project-architecture.md`
- 如果架构发生变化但未更新架构图，不进入交付阶段

但目前流程中还缺少一个显式收口动作：每次功能开发接近结束时，都应主动检查一次“本次是否需要更新架构文档”，而不是只在实现过程中被动判断。

这会导致两类风险：

- 实现已经完成，但没人显式复核架构图是否需要同步
- 即使最终判断“不需要更新”，也没有留下明确结论

因此本轮补充一条标准工作流规则：每次功能开发结束后，必须检查是否需要更新架构文档，并在交付时明确说明检查结果。

## 需求摘要

- 更新 `AGENTS.md`
- 新增规则：每次功能开发结束后，必须显式检查是否需要更新 `docs/architecture/project-architecture.md`
- 如果需要更新，则必须在进入交付阶段前完成更新
- 如果不需要更新，也要在交付说明中明确写出检查结论
- 该规则应进入标准工作流与阶段规范，而不是只作为备注存在

- `lesson01` 强调从入口、应用调度和整体结构理解项目，因此协作规则也应保证架构文档持续与代码现状对齐
- 本轮不改运行时代码，不改变模块依赖方向

## 技术方案与关键数据流

计划在 `AGENTS.md` 中补充两类内容：

1. 在标准工作流中增加“功能结束后检查架构文档是否需要更新”的明确要求
2. 在交付/等待 review 相关规则中增加“无论是否更新，都要说明检查结论”的要求

## 风险点与兼容性影响

- 规则如果写得过于宽泛，执行时仍然容易被忽略，因此需要使用“必须检查”“必须说明结果”这类明确措辞
- 本轮只改协作规范，不涉及运行时代码与项目架构实现
- 一般不需要更新 `docs/architecture/project-architecture.md`，因为本次没有改运行时结构
